## Why this matters

These orchestrators were part of probe turns that Teams already considered complete (bot replied, `correlate_runtime` eventually went to `activeTurns: 0` for some of them) but the Durable Functions instances were never terminated. This is a silent resource leak:

1. **Zombie overseers** — once a turn replies, the overseer instance should wind down. Seven overseers from a single afternoon's probes are all still marked Running.
2. **Zombie `@swarmchatroom` entities** — #680 supposedly closed the "destroy on every exit path" fix, but two swarmchatroom entities from 18:10:03 and 18:30:58 are still live. This is either a regression or the previously-claimed C4 proof on #680 was incomplete — on re-read, my proof bundle only verified `activeTurns: 0` in `correlate_runtime`, which is a DIFFERENT observability surface from Durable Functions instance state.
3. **Cross-contamination risk** — stale overseers may re-activate on scheduler replay and leak outputs into newer turns (this is exactly the failure mode #670 described for pending-intents; session leaks are a broader surface of the same class).
4. **Session count inflation** — TOTAL 357 sessions is unsurprising given the probe volume, but ACTIVE 7 is not normal — it should be 0 or 1 between probes.

## Root cause (hypothesized)

- `#680` fix destroys the entity inside `swarmOrchestrator.ts`'s finally block, but if `swarmOrchestrator` itself hangs before reaching `finally` (which is exactly what #681 / #683 describe), the entity never gets the destroy signal.
- Overseer instances that are waiting on a sub-orchestrator that is itself stuck never release their Running state even after the outer bot reply has been emitted through other channels.
- No Durable Functions timer-based reaper exists to terminate orchestrations that exceed a max-age threshold.
